Làm chủ AI Vibe Coding với Claude Code: Giữ Quyền Kiểm soát khi Con người là "Cửa ải Cuối"

Bài tổng hợp nghiên cứu chia sẻ trên group · lưu vào Radar ngày 2026-07-07 · không rõ tác giả/công cụ tạo gốc (có dạng một deep-research report, trích dẫn nguồn cụ thể: Anthropic, Veracode, CodeRabbit, GitClear, Karpathy...) · ≈ 20–25 phút đọc · toàn văn giữ nguyên, chỉ trình bày lại

TL;DR (nguyên văn từ bài gốc)
  • Con người vẫn là người ra quyết định, không phải người gõ code. Dữ liệu của Anthropic từ ~400.000 phiên Claude Code (10/2025–4/2026) cho thấy con người ra khoảng 70% quyết định lập kế hoạch (làm gì) nhưng chỉ 20% quyết định thực thi (viết code thế nào) — và người có chuyên môn sâu càng nhận về nhiều giá trị hơn. Vai trò của bạn là "gatekeeper": chốt kế hoạch, kiến trúc và review, chứ không đọc mù từng dòng AI sinh ra.
  • Kiểm soát khối lượng bằng quy trình, không bằng sức người. Áp dụng vòng lặp "Explore → Plan → Code → Commit" với Plan Mode, chia nhỏ task, dùng CLAUDE.md/ADR làm "tài liệu sống", và dựng các cổng human-in-the-loop (duyệt plan, PR nhỏ, CI gates, review bằng subagent/second-opinion AI). AI sinh code nhanh gấp bội nhưng khả năng thẩm định của con người mới là nút thắt — phải quản trị "backpressure".
  • Kỹ năng cần nâng cấp là tư duy reviewer/architect. Chuyển từ "coder" sang system design, đọc & review code, testing/eval, và context engineering. Đây là điều Karpathy gọi là chuyển từ "vibe coding" sang "agentic engineering" — nghề nghiệp có kỷ luật, nơi bạn điều phối các agent nhưng vẫn chịu trách nhiệm cuối cùng về chất lượng, bảo mật và kiến trúc.

Key Findings

1Phân công lao động rõ rệt: con người quyết định "cái gì", AI quyết định "thế nào"

Nghiên cứu "How Claude Code is used in practice" của Anthropic (phân tích ~400.000 phiên từ ~235.000 người, 10/2025–4/2026) khẳng định verbatim:

"On average, people make about 70% of the planning decisions but only 20% of the execution decisions... people decide what to build, and the agent decides how to build it."

Càng có chuyên môn, AI càng làm được nhiều việc mỗi lệnh, và tỷ lệ thành công (đo bằng bằng chứng verify được như test pass/commit) càng cao — nghĩa là chuyên môn của bạn là "bộ khuếch đại", không phải thứ bị thay thế.

2Chất lượng code AI có rủi ro đo lường được — code trông sạch nhưng nợ kỹ thuật ẩn

  • Veracode 2025 GenAI Code Security Report (>100 LLM, 80 tác vụ): "45 percent of the code samples failed security tests and introduced OWASP Top 10 security vulnerabilities." Java rủi ro nhất (72% thất bại), XSS thất bại 86% (CWE-80), log injection 88% (CWE-117). Bản cập nhật Spring 2026 của Veracode cho thấy tỷ lệ đạt an ninh vẫn "stuck at approximately 55%".
  • CodeRabbit "State of AI vs Human Code Generation" (17/12/2025; 320 PR AI-đồng tác giả + 150 PR người): PR có AI trung bình 10,83 vấn đề/PR so với 6,45 của người (~1,7x). Lỗi critical +40%, major +70%; ở phân vị 90 là 26 vấn đề/PR (>2x người). David Loker (Director of AI, CodeRabbit): "AI coding tools dramatically increase output, but they also introduce predictable, measurable weaknesses that organizations must actively mitigate."
  • GitClear "AI Copilot Code Quality" 2025 (211 triệu dòng, 2020–2024): code churn (dòng bị sửa/revert trong 2 tuần) tăng từ 3,1% (2020) lên 5,7% (2024), tiếp tục lên ~7,1% (2025); dòng copy/paste lần đầu vượt dòng refactor (8,3%→12,3%).

3Plan Mode là công cụ kiểm soát số một

Đây là chế độ read-only nơi Claude đọc codebase, đặt câu hỏi, và trình bày kế hoạch (file nào sửa, theo thứ tự nào, vì sao) trước khi động vào code. Boris Cherny (tác giả Claude Code tại Anthropic) tự dùng workflow: bắt đầu ở Plan Mode, tinh chỉnh kế hoạch qua lại, rồi mới chuyển sang Auto-Accept để thực thi.

4CLAUDE.md là "điểm đòn bẩy cao nhất" nhưng phải ngắn

Đây là file Claude đọc đầu mỗi phiên. Đồng thuận cộng đồng: giữ dưới ~200–300 dòng (root CLAUDE.md của HumanLayer <60 dòng), chỉ chứa thứ đúng xuyên suốt mọi phiên. File quá dài gây "context rot" khiến Claude bỏ qua chính các quy tắc của bạn.

5Review AI-generated code phải khác review code người

Vì code AI "trông đúng" (plausible) nhưng lỗi tinh vi hơn, review phải mất nhiều thời gian mỗi dòng hơn, tập trung vào: edge cases, concurrency, input validation, dependency ảo (slopsquatting), và việc AI làm yếu CI (xóa test, thêm || true). Dùng nhiều lớp filter: automated gates trước, rồi con người tập trung vào logic/kiến trúc/rủi ro cao. Tại Microsoft, AI review lớp đầu cho >90% PR (>600K PR/tháng), giúp giảm 10–20% thời gian hoàn thành PR — nhưng con người vẫn chốt cuối.

6Anti-pattern nguy hiểm nhất: "trust-then-verify gap" và "accept-without-read"

Karpathy (bài đăng X ngày 26/1/2026) mô tả các agent:

"đưa ra giả định sai mà không kiểm tra, phức tạp hóa code và API, bloat abstraction, sửa/xóa code lân cận không được phép... không quản lý sự bối rối của chúng, không hỏi làm rõ, không nêu ra bất nhất, không trình bày trade-off, không phản biện khi cần."

Details

1. Vấn đề #1: Kiểm soát tốc độ và khối lượng code/tài liệu AI sinh ra

Nguyên tắc nền tảng: "backpressure". Khi năng lực sinh code vượt xa năng lực thẩm định của con người, bạn phải chủ động giới hạn throughput — giống backpressure trong hệ phân tán. Đừng để AI viết cả khối lớn rồi mới review; hãy ép nó đi từng bước nhỏ mà bạn theo kịp.

Workflow lõi: Explore → Plan → Code → Commit (chính thức từ Anthropic).

  • Explore: Vào Plan Mode (Shift+Tab hai lần, hoặc claude --permission-mode plan). Yêu cầu Claude đọc các file liên quan và không viết code. Với codebase lớn, giao việc khám phá cho subagent để không làm đầy context chính: "use a subagent to investigate how our auth system handles token refresh."
  • Plan: Yêu cầu Claude lập kế hoạch chi tiết. Dùng từ khóa "think" / "think hard" / "ultrathink" để cấp thêm ngân sách suy luận. Nhấn Ctrl+G để mở plan trong editor và sửa trực tiếp trước khi duyệt.
  • Code: Thoát Plan Mode, để Claude thực thi từng bước, verify liên tục.
  • Commit: Yêu cầu Claude commit với message mô tả và tạo PR.

Anthropic nhấn mạnh: bỏ qua Explore + Plan là lỗi phổ biến nhất khiến Claude "nhảy thẳng vào code" và giải sai vấn đề. Nhưng Plan Mode cũng có overhead — với task nhỏ (sửa typo, thêm log, đổi tên biến) thì làm thẳng. Quy tắc: nếu mô tả được diff trong một câu, bỏ qua plan.

Chia nhỏ task & incremental development. Dùng stacked PRs (PR nhỏ, atomic, xếp chồng): PR 200 dòng review được, PR 2.000 dòng thì không. PR nhỏ cũng cho AI reviewer kết quả tốt hơn nhiều.

Duy trì mental model về codebase. Dùng chính Claude Code như công cụ onboarding — hỏi nó "logging hoạt động thế nào?", "làm sao thêm API endpoint mới?", "edge case nào mà CustomerOnboardingFlowImpl xử lý?". Tại Anthropic đây đã thành quy trình onboarding cốt lõi. Điểm mấu chốt: bạn phải chủ động hỏi để hiểu, không phải đọc mù toàn bộ diff.

Quản lý context aggressively (đây là ràng buộc cơ bản nhất). Hiệu năng LLM giảm khi context đầy. Do đó:

  • /clear giữa các task không liên quan ("kitchen sink session" là anti-pattern).
  • /compact <instructions> để nén có chọn lọc (vd /compact Focus on the API changes).
  • Sau 2 lần sửa mà vẫn sai → /clear và viết lại prompt tốt hơn, vì context đã bị ô nhiễm bởi các cách tiếp cận thất bại.
  • Dùng /rewind / checkpoint để quay lại trạng thái trước (lưu ý: chỉ track thay đổi của Claude, không thay thế git).

Tài liệu sống (living documentation) để theo kịp:

  • CLAUDE.md — chạy /init để tạo khung, rồi tự tay tinh chỉnh. Cấu trúc tốt: mô tả 1 dòng về project → tech stack → commands (build/test/lint — phần ROI cao nhất) → architecture (3–5 thư mục quan trọng, trỏ tới file thay vì mô tả dài) → conventions → gotchas. KHÔNG lặp lại thứ linter đã enforce. Mẹo vàng (Boris Cherny): mỗi khi Claude làm sai, bảo nó "Update your CLAUDE.md so you don't make that mistake again" — file trở thành vòng phản hồi tự học. Đặt được ở nhiều cấp: ~/.claude/CLAUDE.md (toàn cục), ./CLAUDE.md (check vào git chia sẻ team), ./CLAUDE.local.md (cá nhân, gitignore), và phân cấp cho monorepo.
  • ADR (Architecture Decision Records) — ghi lại quyết định + bối cảnh + lý do + phương án bị loại + hệ quả. Trong codebase AI, ADR trở thành "bộ nhớ dự án bền vững" cho cả người và AI: nó ngăn AI đề xuất lại phương án team đã cân nhắc và loại bỏ (vd đề xuất tách microservice khi team đã chọn modular monolith). Có thể nhờ Claude quét codebase sinh ADR theo template MADR — nhưng phải review phần "Context" và "Consequences" vì AI có thể bịa ra cái "why".
  • Skills (.claude/skills/*/SKILL.md) cho kiến thức chỉ dùng đôi khi (deploy, SEO workflow) — nạp theo yêu cầu, không làm phình context mỗi phiên. Giữ SKILL.md dưới 500 dòng.

2. Vấn đề #2: Nắm và đánh giá kiến trúc, làm "cửa ải cuối"

Yêu cầu AI đưa nhiều phương án kèm trade-off. Đừng chấp nhận một kiến trúc duy nhất. Prompt mẫu:

"Tạo bảng trade-off so sánh 3 phương án cho [vấn đề]. Tiêu chí: chi phí, độ nhất quán, tốc độ failover, độ phức tạp vận hành, tuân thủ. Nêu rõ ưu/nhược mỗi phương án và khuyến nghị đường ít rủi ro nhất."

Chi phí hỏi là 2 phút hội thoại; chi phí chọn sai là nhiều tuần refactor.

Áp dụng tư duy ATAM (Architecture Tradeoff Analysis Method) đơn giản hóa. Đánh giá kiến trúc theo quality attributes (scalability, security, maintainability, cost), xác định risks, sensitivity points và trade-offs, ánh xạ ngược về yêu cầu. Nghiên cứu học thuật xác nhận con người muốn AI đóng vai "design analyzer" phân tích trade-off/risk — nhưng cũng khẳng định:

"AI should not settle the final decision. Because this is the part that people should be accountable for."

Hỏi ngược lại AI & để AI phỏng vấn bạn. Với feature lớn, dùng:

"I want to build [X]. Interview me in detail using the AskUserQuestion tool. Ask about technical implementation, edge cases, tradeoffs... then write a complete spec to SPEC.md."

Điều này ép làm rõ trước khi code, và tạo spec self-contained để chạy ở phiên mới sạch context.

Cross-check bằng AI khác (adversarial / second opinion). Vì một reviewer chạy trong context mới chỉ thấy diff + tiêu chí (không thấy lý luận đã tạo ra code) nên đánh giá khách quan hơn. Các cách:

  • Writer/Reviewer pattern: Session A viết code, Session B (context sạch) review — không bị thiên vị vì "code mình vừa viết".
  • Subagent review: "use a subagent to review this code for edge cases" hoặc chạy skill /code-review có sẵn (review diff hiện tại trong subagent mới, trả findings về phiên chính).
  • Cross-model: Plugin codex của OpenAI gọi từ trong Claude Code để lấy second opinion từ model khác; plugin claude-review-loop (Hamel Husain) tự động chạy Codex review qua Stop hook trước khi bạn chấp nhận thay đổi.
  • Cảnh báo: reviewer được yêu cầu "tìm lỗ hổng" sẽ luôn báo cáo một số, kể cả khi code ổn. Bảo nó chỉ flag gap ảnh hưởng correctness/yêu cầu, tránh over-engineering (thêm abstraction thừa, code phòng thủ, test cho case không thể xảy ra).

Quy trình review code AI-generated hiệu quả (checklist 10 điểm):

  1. Requirement fulfillment: có đủ acceptance criteria không?
  2. Edge cases: null, empty list, dữ liệu sai định dạng, hành vi bất thường.
  3. Logical soundness: đi bộ qua logic phức tạp (AI giỏi code "trông đúng").
  4. Error handling: không crash, log context, xử lý exception.
  5. Concurrency: hàm gọi 2 lần liên tiếp thì sao?
  6. Input validation: mọi input ngoài (user/API/file/queue) được validate/sanitize.
  7. Performance: cảnh giác I/O tuần tự trong vòng lặp (AI hay tạo hàng trăm request nối tiếp thay vì song song).
  8. Dependencies: mọi thư viện đề xuất có thật, còn được bảo trì, không lỗ hổng (cảnh giác slopsquatting/package ảo).
  9. Secrets: không hardcode API key/credential.
  10. CI integrity: AI KHÔNG được làm yếu CI (xóa test, || true, skip lint) — đây là blocker tuyệt đối.

Thiết lập human-in-the-loop gates (nhiều lớp filter):

  1. Plan approval gate: không code khi plan chưa được duyệt.
  2. Automated pre-review gates (chạy trước khi người xem): lint, format, typecheck, unit test, static analysis (vd Snyk/CodeQL), secret scanning.
  3. Hooks (deterministic, không thể hallucinate): PreToolUse chặn lệnh nguy hiểm / ghi vào thư mục cấm; PostToolUse chạy linter sau mỗi edit; Stop hook ép chạy check cuối phiên. Hooks đảm bảo "luôn xảy ra" khác với CLAUDE.md chỉ mang tính khuyến nghị (Claude tuân ~80%). Có loại hook agent/prompt để spawn subagent kiểm tra logic phức tạp.
  4. PR review gate: con người duyệt cuối. Mô hình Microsoft: AI làm lớp đầu >90% PR (>600K PR/tháng), con người vẫn chốt.
  5. CI/CD & test coverage gate: coverage là hợp đồng với hành vi hiện tại; không merge khi test đỏ.
  6. /goal condition & verification: cho Claude một check chạy được (test/build/screenshot) để nó tự đóng vòng lặp; yêu cầu nó trưng bằng chứng (output test) thay vì khẳng định "đã xong". Đây là điều Anthropic gọi là practice hiệu quả nhất: "Give Claude a way to verify its work."

Trách nhiệm người approve. Karpathy (Sequoia Ascent 2026, ngày 30/4/2026) nói thẳng:

"You are not allowed to introduce vulnerabilities because of vibe coding. You are still responsible for your software, just as before."

— và tóm gọn:

"Vibe coding raises the floor. Agentic engineering raises the ceiling."

Bốn câu hỏi phải trả lời được trước khi merge: code làm gì, tại sao làm thế, hỏng ở đâu, ai chịu trách nhiệm. Nếu không trả lời được → code chưa được hiểu → chưa được sở hữu → không merge.

3. Vấn đề #3: Lộ trình upskill và cách làm việc đúng

Từ "vibe coding" sang "agentic engineering" (Karpathy). Vibe coding = mô tả, nhận code, chấp nhận, đi tiếp — nâng "sàn" (ai cũng làm được) nhưng không có "thanh chất lượng". Agentic engineering = nghề có kỷ luật, nâng "trần": bạn điều phối các agent stochastic nhưng vẫn giữ conceptual ownership, review mọi diff về kiến trúc (không chỉ syntax), xây eval loop, giữ ranh giới bảo mật. Karpathy: ở tầng cao, "technical mastery là bội số còn lớn hơn trước" — người hiểu sâu kiến trúc đạt 10–100x năng suất, người mới chỉ sinh code hỏng nhanh hơn.

Kỹ năng quan trọng hơn trong kỷ nguyên AI coding (thứ tự ưu tiên):

  1. System design & architecture — vì AI giỏi local optimization nhưng không hiểu hệ quả cấp hệ thống. Đây là "taste" mà senior có nhờ đã đi qua đủ hệ thống thực.
  2. Code reading & review — kỹ năng đọc diff tìm lỗi conceptual, cross-system assumptions (không chỉ syntax).
  3. Testing & eval design — xây feedback loop (test/eval/benchmark) để agent tự cải thiện; theo Karpathy, "traditional software tự động hóa cái specify được, LLM tự động hóa cái verify được."
  4. Context engineering — được cộng đồng gọi là kỹ năng chuyển dịch quan trọng nhất 2026: cấu trúc CLAUDE.md, skills, hooks, chọn context, quản lý memory. Sâu hơn prompt engineering.
  5. Spec design — viết spec chi tiết (invariants, security boundaries) trước khi prompt.
  6. Domain expertise & judgment — dữ liệu Anthropic: chuyên môn càng cao, tỷ lệ thành công càng cao.

4. Áp dụng theo bối cảnh

(a) Dự án công ty, codebase lớn, làm việc nhóm:

  • CLAUDE.md check vào git để cả team đóng góp; dùng CLAUDE.md phân cấp (root + thư mục con) cho monorepo.
  • Coding standards cho AI-generated code: nhúng policy ở 3 nơi — CONTRIBUTING.md, PR template (kèm checklist 10 điểm), và file instruction của AI (CLAUDE.md). Yêu cầu PR description ghi rõ: AI đóng vai gì, prompt/context nào, con người đóng góp/verify gì.
  • Review policy phân theo rủi ro: thay đổi low-risk đi nhanh; thay đổi API/auth/dữ liệu nhạy cảm/AI-generated nhận review sâu hơn. Đừng để thời gian review PR-AI bằng PR-người — nếu bằng nghĩa là đang "rubber-stamp".
  • Phân phối subagents/hooks/commands qua plugins để cả team cùng chuẩn (vd cài code-review plugin 4-agent của Boris Cherny, gán điểm tin cậy 0–100, lọc dưới 80 để giảm false positive).
  • MCP servers cho dữ liệu nhạy cảm (kiểm soát/log truy cập) thay vì CLI trực tiếp.

(b) Side project / micro-SaaS một người:

  • Cân bằng tốc độ vs bảo trì: dùng auto-accept mode cho phần "rìa" (peripheral, như UI phụ), làm việc đồng bộ với prompt chi tiết cho core business logic. Tại Anthropic, RL team báo cáo auto-accept thành công ngay lần đầu chỉ ~1/3 lần, nên checkpoint thường xuyên để revert dễ.
  • Vẫn giữ tối thiểu: CLAUDE.md tốt, test coverage cho code core, ADR cho quyết định lớn (để "bạn của 6 tháng sau" hiểu được).
  • Coi "domain expertise là nút thắt, không phải công cụ" (bài học từ ranthebuilder.cloud: Claude build website nhanh nhưng SEO/security hardening/analytics phải tự lái vì hiểu tại sao chúng quan trọng).
  • Dùng git worktrees chạy nhiều session song song (feature + bugfix) không đụng nhau.
  • Cảnh giác "90-day reckoning": code vibe nhanh ngày 1 nhưng nợ kỹ thuật lộ ra sau ~3 tháng; phòng bằng test + review kiến trúc + enforce consistency (error handling nhất quán) ngay từ đầu.

Recommendations

Giai đoạn 0 — Thiết lập nền (tuần 1)

  1. Viết CLAUDE.md thủ công (<200 dòng): commands, architecture 3–5 thư mục, conventions, gotchas. Check vào git.
  2. Cấu hình settings.json deny-first: chặn lệnh nguy hiểm. Thêm 1–2 hook cơ bản (lint sau edit, chặn ghi vào migrations/). Thứ tự: quyền → /init CLAUDE.md → tùy chỉnh.
  3. Cài Karpathy-style guardrails vào CLAUDE.md (think-before-coding, no silent assumptions, minimal solution first, no orthogonal edits).

Giai đoạn 1 — Kỷ luật workflow (tuần 2–4)

  1. Mặc định mọi task không tầm thường: Explore (Plan Mode) → duyệt plan → Code → verify → Commit.
  2. Với mọi feature: yêu cầu 2–3 phương án kiến trúc kèm trade-off table trước khi chọn.
  3. Tạo subagent code-reviewer (context riêng, model opus cho task quan trọng) và chạy nó/skill /code-review trên mọi diff trước khi merge.
  4. Dùng /clear giữa các task; sau 2 lần sửa sai thì reset + prompt lại.

Giai đoạn 2 — Cổng & tự động hóa (tháng 2)

  1. Dựng CI gates: lint + typecheck + test + secret scan + static analysis; PR template với checklist 10 điểm và 3 câu hỏi về vai trò AI.
  2. Thêm cross-model second opinion (plugin codex hoặc claude-review-loop) cho các thay đổi rủi ro cao.
  3. Đóng gói workflow team thành plugin để đồng bộ.

Giai đoạn 3 — Upskill reviewer/architect (tháng 3+ và liên tục)

  1. Học system design: Designing Data-Intensive Applications (Kleppmann & Riccomini, bản 2 — 2/2026), ByteByteGo, The System Design Primer (repo miễn phí của Donne Martin), Gaurav Sen (YouTube miễn phí).
  2. Đọc chính thống về Claude Code: docs Anthropic (code.claude.com/docs — best-practices, common-workflows, sub-agents, hooks), báo cáo "How Anthropic teams use Claude Code" và "How Claude Code is used in practice".
  3. Học spec-driven development: thử GitHub Spec Kit (open-source, agent-neutral, các lệnh /speckit.specify → plan → tasks → implement) hoặc BMAD-METHOD trên 1 feature nhỏ.
  4. Đọc "A Philosophy of Software Design", "The Pragmatic Programmer" cho tư duy trade-off/craftsmanship.

Ngưỡng để điều chỉnh (benchmark)

  • Nếu thời gian review PR-AI ≈ PR-người → bạn đang rubber-stamp, tăng review depth.
  • Nếu code churn dự án >7% hoặc bug rate tăng → siết gates, giảm auto-accept.
  • Nếu Claude bỏ qua quy tắc CLAUDE.md → file quá dài, cắt bớt hoặc chuyển thành hook.
  • Nếu AI thành công lần đầu <~1/3 lần cho task autonomous → task chưa đủ scoped, quay lại planning.
  • Nếu bạn không trả lời được "code này làm gì / hỏng ở đâu" → dừng, không merge.

Caveats

  • Số liệu chất lượng code AI đến từ nhiều nguồn với phương pháp khác nhau — Veracode (45% OWASP), CodeRabbit (~1,7x vấn đề), GitClear (churn ~2x) đo các thứ khác nhau, không nên gộp thành một con số. Con số "2,74x" của CodeRabbit chỉ áp dụng cho XSS, không phải toàn bộ security; và cần lưu ý CodeRabbit không thể xác minh 100% rằng nhóm PR "người" hoàn toàn không dùng AI.
  • Tính năng Claude Code thay đổi rất nhanh (Anthropic ship gần như hằng tuần); tên lệnh/phím tắt/model có thể đổi. Hãy bám nguyên tắc (research→plan→execute→review), không bám phiên bản cụ thể.
  • Karpathy guidelines repo (220K+ sao) chưa được Karpathy chính thức endorse; nó thiên về thận trọng, có thể gây over-engineering/friction cho task đơn giản một dòng.
  • Các công cụ/plugin bên thứ ba (Graphite, CodeRabbit, Cursor Bugbot, các skill cộng đồng) là gợi ý, không phải khuyến nghị tuyệt đối; đánh giá theo bối cảnh, compliance và chi phí token của bạn. Nhiều bài blog là nguồn thứ cấp/tiếp thị — ưu tiên docs Anthropic và báo cáo nghiên cứu gốc.
  • Chi phí token: spec-driven & multi-agent tốn token hơn ~20–40%; theo dõi bằng /cost và dùng .claudeignore để loại file không cần đọc.
